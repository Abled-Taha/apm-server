Endpoint Documentation

# Endpoint Description
1 / The Index page, it shows you an alert if the server is running correctly otherwise just a 404 page is returned.
2 docs/ The Documentation page, it shows you a table of all the endpoints of this API which can be accessed by a request and describes them. It also shows some other usefull information.
3 signup/ The SignUp page, If the request made to this endpoint is of type POST then it validates the credentials in the request and if everything is alright then it creates the user account. And if the request made to this endpoint is of type GET then it returns an HTML page with a form which needs to be filled with credentials which then are validated and if everything is alright then it creates the user account. But, if the the type of request is neither POST nor GET (PUT, etc) then it returns an HTML page which tells the user that only POST and GET requests are allowed to this endpoint.
4 login/ The Login page, if the request made to this endpoint is of type Post then it validates the credentials in the request and if everything is alright then it logs the user in. And if the request made to this endpoint is of type GET then it returns an HTML page with a form which needs to be filled with credentials which then are validated and if everything is alright then it logs the user in. But, if the type of request is neither POST nor GET (PUT, etc) then it returns an HTML page which tells the user that only POST and GET requests are allowed to this endpoint.

Config File


Django Secret Key

Django uses a secret key, which can be found in the config.json file of this project and the default one should not be used in any case. To generate a new django secret key, you can use any online generator like DJECRETY. The other method to generate a secret key is by command line which I would not be covering.


Debug Mode

This is a Boolean (True / False) value which can be set in the project's config.json file. It needs to be set to FALSE for normal use. If set to TRUE, it may help you debug the application but shows many things which should be public in any case. So, other than when running the API in a development environment, the value should be set to FALSE.


Supply GUI Response

This is a Boolean (True / False) value which can be set in the project's config.json file. If the API should not allow any GET requests then this variable should be set to False otherwise it can be left to True. When set to True the API will also respond to GET requests and return HTML pages.


Host DB

This is a variable of type STRING, it contains the IP of the host on which the MongoDB is running. If the database is running on the same machine as the API, the value of it can be set to 'localhost' or '127.0.0.1'.


Port DB

This is a variable of type STRING, it contains the PORT on which the MongoDB is listening. The default port for MongoDB is '27017'. If the default port for the database is left untouched when setting it up, this variable can be left untouched also or be set to '27017'.


Name DB

This is a variable of type STRING, it contains the name of the dabase in a MongoDB cluster. If the exact instructions are followed from the README.md file of this project then this variable can be left untouched.


Username

This is a variable of type STRING, it contains the name of the user which has rights to READ & Write to the database. If the exact instructions are followed from the README.md file of this project then this variable can be left untouched.


PWD

This is a variable of type STRING, it contains the password of the user which accesses the database. The password is set when creating this user. If the README.md file has been followed then the password which had been set at the time of setting up the user should be set in this field also.
